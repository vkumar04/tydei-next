## Why this file exists

Charles reported (iMessage, 2026-04-20 11:13 AM): *"rebate earned vs
collected not matching on the list screen vs in the contract."*

The columns labeled "Rebate Earned (YTD)" and "Spend (Last 12 Months)"
on the contracts list page (`components/contracts/contract-columns.tsx`
L304-353) are declared with **dual-source fallbacks**:

```ts
// Spend column (L308, L320)
accessorFn: (row) => row.currentSpend ?? row.metricsSpend ?? 0
// Earned column (L330, L345)
accessorFn: (row) => row.metricsRebate ?? Number(row.rebateEarned ?? 0)
```

The two sources are:

- `rebateEarned` / `rebateCollected` / `currentSpend` — computed
  in-memory by `getContracts` in `lib/actions/contracts.ts` via the
  canonical helpers `sumEarnedRebatesYTD`, `sumCollectedRebates`, and
  the trailing-12mo spend cascade.
- `metricsRebate` / `metricsSpend` — computed Prisma-side by
  `getContractMetricsBatch` in the same file (L627+), with an explicit
  "keep in sync" comment at L697.

The contract **detail** page always reads the canonical helpers. The
contract **list** page's column accessor prefers `metricsRebate` /
`metricsSpend` whenever present. When the two reducers disagree (and
the "keep in sync" comment doesn't actually keep them in sync — it's a
comment, not a constraint), the column overrides the canonical value.
Charles's screenshot is that override materializing.

## Structural drift modes

Without running the script live we can still enumerate the ways the
two paths go out of sync, because they disagree on algorithm:

| Drift mode | Canonical path | Batch path | Effect |
|---|---|---|---|
| YTD cutoff | `sumEarnedRebatesYTD` iterates in JS after the DB query, so any rebate where `payPeriodEnd` has passed AND lies in the current calendar year is counted | Uses a DB `WHERE payPeriodEnd gte startOfYear AND lte today` on `Rebate.groupBy` | Matches in steady state, but diverges when rebate rows are seeded with sub-second `payPeriodEnd` timestamps or when the in-memory helper includes a row with `payPeriodEnd === startOfYear` that the DB range excludes (range boundary semantics differ) |
| Spend window | `getContracts` uses a trailing-12mo window (`today - 1 year` .. `today`) and a 3-step cascade (ContractPeriod → COG-by-contract → COG-by-vendor+category) | `getContractMetricsBatch` uses **lifetime** COG-by-contract with no trailing-window filter, then falls back to ContractPeriod (also lifetime) | Any contract with >12mo of COG data will show a higher batch spend than canonical spend, so the list shows a different $ than the detail header |
| Rebate filter | Canonical counts only closed periods (`payPeriodEnd <= today`) | Batch also counts closed periods but via DB aggregate that doesn't join `collectionDate`, matching canonical — but this is a coincidence of current implementation | Today matches; any future edit to either side breaks parity silently |
| Facility scope | Canonical joins via `contractsOwnedByFacility` (multi-facility contracts OK) | Batch scopes by `facilityId` on the `Rebate`/`ContractPeriod` rows themselves (already fixed in QA list-2) | Matches, but duplication of the scope means a regression on either side skews list vs detail |

## The column-accessor bug

The largest visible delta is the **spend window mismatch**. The batch
returns lifetime COG-by-contract spend; the canonical helper returns
trailing-12mo spend; the column's `?? metricsSpend` fallback displays
the lifetime number when the trailing-12mo figure is $0 (e.g., a
contract whose window starts a few months ago with no enrichment yet).
For Charles's contract with $4,467,188 contract value and $1,536,659
reported spend, the delta is the 12-month vs lifetime wedge.

The `metricsRebate` shadow is more subtle: for contracts whose
`rebateEarned` is $0 in `getContracts` (e.g., no seeded `Rebate` rows
yet), the accessor falls back to the batch, which may return a
`ContractPeriod.rebateEarned` rollup. Those rollups are frozen
snapshots of historical accrual and can be *different* from what the
canonical helper computes at read time. For Charles the reported
canonical YTD is ~$58,660 and the batch figure differs.
